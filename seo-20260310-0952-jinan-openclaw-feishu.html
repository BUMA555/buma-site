Official Facts

先把官方资料里能确认的关键信息讲清楚

这部分不是猜测，都是 OpenClaw 官方文档已经明确写出来的内容。

OpenClaw 是什么

官方入门文档把它定位为自托管 Gateway，用来把聊天渠道接进 AI 助手。最快的本地验证方式是 `openclaw dashboard`，默认可在本机浏览器打开控制台。

环境要求

官方 Getting Started 写得很明确：推荐 Node 24，Node 22 LTS 也兼容。第一步不是写提示词，而是先把 `openclaw onboard`、`openclaw gateway status` 这些基础动作跑通。

飞书接入方式

官方 Channels 文档明确列出 Feishu/Lark；Feishu 文档说明它通过平台 WebSocket 事件订阅接入，这意味着你不一定非得先暴露公网 webhook 才能开始。

Real Workflow

按这个顺序做，飞书接入成功率更高

真正容易出错的不是步骤多，而是顺序错。下面是更接近实战的落地顺序。

第 1 步：先把 OpenClaw 网关跑起来

  • 安装后先跑 `openclaw onboard`，让向导把基础配置写好。
  • 随后执行 `openclaw gateway status`，确认网关已经起来。
  • 再用 `openclaw dashboard` 或本地控制台确认最基本的聊天能力可用。

这一层没通，后面飞书事件订阅即便填完也大概率收不到消息。

第 2 步：去 Feishu 开应用并拿凭证

  • 进入 Feishu Open Platform 新建企业应用。
  • 在 Credentials & Basic Info 里拿到 App ID 和 App Secret。
  • 如果你用的是国际版 Lark，官方文档提醒要额外设置 `domain: \"lark\"`。

这一步最重要的是保管好 App Secret，不要乱散发。

第 3 步：权限别乱猜，按文档开

  • Feishu 文档给了可批量导入的 scopes 列表，包括消息读取、发送、资源访问等权限。
  • 至少要关注 `im:message`、`im:message:readonly`、`im:message:send_as_bot` 这类核心权限。
  • 没批对权限，最常见现象就是应用能建起来，但机器人不收消息、不回消息。

这一步不要省，很多人就是卡在权限没配齐。

第 4 步：先开 Bot，再配事件订阅

  • 先启用 Feishu 应用里的 Bot 能力并设置机器人名称。
  • 官方文档强调：配事件订阅前，要先完成 `openclaw channels add` 且保证网关正在运行。
  • 在 Event Subscription 里选择“Use long connection to receive events”，并添加 `im.message.receive_v1`。

如果网关没开就去配长连接，保存失败或者后续不生效，是非常常见的问题。

最常见的 5 个坑

  1. 网关没启动就去配事件订阅，导致 Feishu 长连接保存失败。
  2. 只建了应用，没启用 Bot 能力，结果消息入口根本没打开。
  3. 权限没按文档导入完整，机器人无法读写消息。
  4. 把本该先在 Control UI 验证的基础问题，拖到飞书接入后才排查。
  5. 国际版 Lark 没设 `domain: \"lark\"`，结果域名环境不一致。

哪些业务最适合先接这条链路

  • 咨询量不大，但问题重复度很高的服务型业务
  • 内容生产、FAQ 回复、资料归档已经压得人很累的一人公司
  • 想先试运行 1 个机器人，再逐步加岗位分工的小团队

不建议一上来就追求“十几个 Agent 全自动联动”。先把飞书消息进入、分流、应答跑顺，收益更直接。
