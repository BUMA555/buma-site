OpenClaw 飞书长连接怎么接：WebSocket 事件订阅与 Pairing 审批清单

如果你在搜 OpenClaw 飞书接入、飞书 WebSocket 长连接、事件订阅 im.message.receive_v1 或 pairing 审批，基本说明你已经走到了“能不能稳定收消息”的阶段。这篇内容基于 OpenClaw 官方 Feishu/Pairing/Getting Started 文档与 Clawdbot 中文文档整理，目标不是堆概念，而是帮你把三件关键事做清楚：飞书事件订阅要选哪种模式、OpenClaw 网关必须先跑起来、以及 pairing 审批到底是用来保护谁的。

Quick Judgment

先说结论：飞书接不稳，80% 不是“机器人坏了”，而是“事件订阅或网关顺序没配对”

OpenClaw 文档明确强调：飞书通道通过 WebSocket 事件订阅接收消息，前提是网关已经启动；pairing 是所有陌生私聊的安全闸门，不审批就不会处理消息。把这两条理解清楚，就能少走大半弯路。

01

先有网关，再配事件

无论是向导还是 CLI，官方都建议先完成 OpenClaw 安装与网关启动，再去飞书后台配置事件订阅。网关没跑起来，长连接模式保存就容易失败。

02

长连接模式是默认推荐

Feishu 通道文档明确说明使用 WebSocket 长连接接收事件，这样可以不暴露公网回调 URL，更适合一人公司与内网场景。

03

Pairing 是安全闸门

Pairing 文档说明：未知用户会收到配对码，未审批前消息不会被处理。它不是“多一步麻烦”，而是避免陌生人直接接入你的机器人。

OpenClaw 飞书 WebSocket 接入的最小闭环（6 步）

1. 跑通 OpenClaw 基础环境

OpenClaw Getting Started 文档建议先完成安装与网关启动，常用命令是 openclaw onboard --install-daemon 和 openclaw gateway status。这一步是后续所有飞书接入的底座。

2. 创建飞书企业自建应用

中文文档与官方说明一致：在飞书开放平台创建企业自建应用，获取 App ID 与 App Secret，并先把机器人能力打开。

3. 配置权限与事件

文档提供了批量导入权限的 JSON 示例；事件订阅需要选择“使用长连接接收事件（WebSocket）”，并添加 im.message.receive_v1 等消息事件。

4. 在 OpenClaw 中添加 Feishu 渠道

运行 openclaw channels add，选择 Feishu，填入 App ID 与 App Secret。完成后重启或检查网关日志，确认通道已加载。

5. 测试消息 + Pairing 审批

飞书里给机器人发消息后，若默认 dmPolicy 为 pairing，会收到配对码。使用 openclaw pairing list feishu 与 openclaw pairing approve feishu <CODE> 完成审批。

6. 收口安全边界

如果你只允许特定人私聊，可用 allowlist；若在群聊里要求必须 @ 提及，也可通过群组策略与 requireMention 控制。

Pairing 机制到底在保护什么？

Pairing 文档的核心意思是：OpenClaw 不会默认信任任何陌生私聊。首次发消息会生成 8 位配对码，且配对请求是有时效与上限的。这一步是为了确保你能明确批准“谁可以和机器人说话”。

对一人公司来说，这意味着你不用担心把机器人暴露给陌生人；对小团队来说，它相当于一个轻量的“人群门禁”。只要你没批准，消息不会进入真正的处理链路，避免了无权限人员触发成本与数据泄露。

如果业务需要更严格的管控，也可以启用 allowlist，让机器人只回应被允许的 Open ID。这样你可以把“谁能聊”收口到非常明确的名单里。

Common Pitfalls

飞书接入最容易踩的 4 个坑

这些问题不是“飞书坏了”，而是顺序与边界没对齐。

事件订阅配置在网关之前

文档明确提醒：如果网关未启动，长连接模式可能保存失败。先起网关再配事件，是最稳的顺序。

忘了加 im.message.receive_v1

事件订阅里没添加消息接收事件，机器人就“像离线”。这一步经常被忽略，导致误判为配置失败。

配对码没审批就期待回复

pairing 的默认策略是先拦截再审批。没批准之前，消息不会进处理链路，这是安全策略，不是 bug。

群聊策略没设置

群聊里如果要求必须 @ 触发，记得把 requireMention 配好。否则你会看到“机器人不回话”的假象。
