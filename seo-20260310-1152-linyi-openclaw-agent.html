01

OpenClaw 是自托管 Gateway

官方文档明确将 OpenClaw 定义为 self-hosted gateway，用来连接聊天渠道、会话与 Agent 能力，核心价值是把多入口整合到同一控制层。

02

官方推荐先走 onboard 向导

公开入门资料推荐使用 openclaw onboard 完成安装与基础配置，并通过 Control UI 查看聊天、配置、session 和状态。

03

支持多渠道，也包含 Feishu

GitHub 公开说明和文档都把 Feishu 列入支持通道范围，但是否适合你的业务，还要结合权限、路由规则和实际协同方式来配置。

Workflow Order

临沂做 OpenClaw Agent 工作流，建议先按这 4 步推进

先把一个闭环跑通，再逐步扩岗位，比一次性铺很多功能更适合一人公司。

第一步：先确认 Gateway 能稳定运行

公开资料写明运行环境要求 Node 22+。先完成 onboard、确认 gateway 状态和本地控制界面可用，再去谈业务流程，否则后面的工作流会很虚。

第二步：只选一个核心消息入口

不要一开始把所有通道都接上。对很多临沂团队来说，先围绕 Feishu 或最常用咨询入口打通一条线，比多渠道同时开工更容易出结果。

第三步：按岗位定义 Agent，不让它们乱抢活

内容生产、客户承接、老板提醒、资料整理分别做什么，哪些内容能自动发，哪些必须人工确认，这一步比提示词花活更重要。

第四步：围绕真实业务持续迭代

工作流不是搭完就结束。后续还要根据 FAQ、页面转化、消息承接效果，持续优化知识内容、话术边界和协同节奏。
