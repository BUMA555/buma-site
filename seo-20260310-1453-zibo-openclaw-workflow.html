Public Facts

先讲公开信息，再谈怎么改写成可落地页面

这篇页只基于 OpenClaw 官网首页、Getting Started、Feishu 渠道文档与 GitHub 仓库公开说明重写，不抄原文，不包装虚假客户案例。

01

OpenClaw 是自托管 Gateway

官网公开介绍它是 self-hosted gateway，可连接 WhatsApp、Telegram、Discord、Feishu 等渠道，并承担会话、路由、工具与控制界面这层底座能力。

02

起步路径很明确

Getting Started 页面公开给出 Node 22+、安装、openclaw onboard --install-daemon、openclaw gateway status 和 openclaw dashboard 这条基础路径，说明它并不是只能“看概念”的项目。

03

Feishu 适合做协同入口

Feishu 渠道文档公开说明它可通过企业应用、Bot 能力和 WebSocket 事件订阅接消息，这让 OpenClaw 更适合接进内部协同，而不只是停在单点聊天。

为什么“淄博 + OpenClaw + 工作流”值得单独布局

这类词更接近执行阶段

搜“工作流”而不是只搜产品名，通常已经在判断怎么接进现有业务，离真正咨询更近。

一人公司更需要顺手的执行链路

一个人做业务，最怕每天都在切消息、补资料、催进度。工作流的价值在于把高频动作串起来，让事情不靠临时记忆推进。

淄博本地业务适合先做小闭环

无论是本地服务、制造配套还是个人品牌项目，先把线索承接、内容协同或内部提醒其中一条流程跑稳，效果比空谈“全自动化”更实在。

哪些淄博场景适合先接 OpenClaw 工作流

咨询线索统一承接

当消息来自官网、微信、飞书或不同人手里时，先做统一入口与提醒，能明显减少漏跟进。

内容生产与审核分工

内容 Agent 先写初稿、整理素材、改写多版本，人工只负责定调和确认，适合一人公司提升输出频率。

资料整理与任务提醒

报价单、FAQ、会议纪要、素材链接这类重复整理工作，很适合先交给 Agent 做补位。

Workflow

淄博 OpenClaw 工作流，建议先按这 4 步推进

先把底座跑通，再接最值钱的一条业务链路，这更符合 BUMA 一贯的落地节奏：赋能、放大、补位、成事。

第一步：先把 Gateway 和控制界面跑起来

公开文档已经给出最短路径。先确认安装、onboard、Gateway 状态和控制界面都正常，再去谈工作流，否则排查成本会很高。

第二步：只选一个最高频场景

建议先从咨询承接、FAQ 回复、内容草稿流转或提醒同步里选一个，不要一开始就做全渠道、全岗位、全流程。

第三步：给不同 Agent 划清岗位边界

内容 Agent 负责生成与整理，协同 Agent 负责提醒和分发，人工负责关键确认。边界一清楚，质量和风险都会更可控。

第四步：围绕真实使用反复迭代

当第一条链路跑稳，再叠加更多知识库、更多渠道或更多岗位 Agent，而不是在第一周就追求大而全。

What To Avoid

淄博团队做 OpenClaw 工作流，最容易踩的 3 个坑

这些坑不是想象出来的，而是从公开资料与实际工作流逻辑里能预判到的共性问题。

一开始就想替代所有人工

涉及报价、承诺、客户关系和最终判断的动作，还是要保留人工把关，Agent 更适合补位和提速。

没先统一消息入口

入口没统一，后面的提醒、分发、归档都很难稳定，工作流容易停在“偶尔能跑”。

功能堆很多，复盘却没人做

真正能长期用下去的工作流，一定是有人复盘、规则会沉淀、场景会逐步收敛，而不是功能表越长越好。
