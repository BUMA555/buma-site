1. 为什么一人公司合并旧文时，redirect 检查不能靠感觉

在 内容合并清单 里，你已经决定了这篇是 keeper page、那篇要并进来、第三篇直接 301 跳走。但"决定做 301"和"做完的 301 真的有效"之间，隔着一整套 QA 检查。

一人公司最容易出现的错误是：合并完了就直接等效果，不知道 redirect chain 可能断了、canonical 可能打错了、Google 可能还在抓旧 URL。这个阶段的损失往往比合并本身更大——你损失的不只是排名，还有本来可以通过合并保住的权重。

Merge Redirect QA Checklist 就是为了解决这个问题：在你按下 redirect 上线之后，有一张清单可以快速扫一遍，确保"跳了"这件事真的发生了、跳对了地方、Google 也知道跳了。

一句话结论

301 跳转不保证 SEO 价值真的传过去——只有被正确执行的 301 才能转移排名信号。合并后不 QA，等于把权重传递交给运气。

3. Merge Redirect QA Checklist 真正负责什么，和 consolidation / relaunch 的边界

在一人公司官网的内容体系中，已经有以下几层：

  • 内容合并清单：负责判断哪篇是 keeper page、哪些 secondary pages 要并入或跳转
  • 内容修剪清单：负责判断哪些页该删、哪些该降权
  • 旧文更新 SOP：负责动手改某篇旧文时的标准执行步骤
  • 内容重启清单：负责改完后如何像新文一样重新上线
  • 旧文更新后分发清单：负责改完后怎么重新送出去
  • 重新索引请求清单：负责判断改完后是否值得催 Google 来抓

Merge Redirect QA Checklist 负责的是：在 content consolidation 决定跳转、上线 redirect 之后，验证 redirect 是否真的执行正确、权重是否真的能传递、Google 是否知道这次跳转。它回答的是合并后 QA 层的 6 个问题：

  1. 第一步检查什么：redirect 上线前，源 URL 和目标 URL 各自状态是否清晰。
  2. 第二步查什么：redirect 响应码是否真的是 301/308，而不是 302/307。
  3. 第三步查什么：redirect chain 是否存在——即源 URL 是否经过多跳才到目标。
  4. 第四步查什么：keeper page 的 canonical 是否指向自己，而不是意外指向被合并的旧页。
  5. 第五步做什么：提交后观察什么指标，以及何时判断是否需要重新提交。
  6. 第六步判断什么：哪些情况不应该 redirect，而是应该 404 直接删除。

4. 可直接执行的 Merge Redirect QA Checklist：六步系统检查

下面这份 checklist，适合一人公司在 consolidation checklist 确认了 keeper page、上线了 redirect 之后，直接照着执行。每一步都有明确的操作和确认方式。

Merge Redirect QA Checklist / 合并重定向质量检查清单

执行前提：已经通过 内容合并清单 确认了 keeper page，并通过 内容修剪清单 判断了哪些该跳转、哪些该 404。

第一步｜Redirect 上线前：确认源 URL 与目标 URL（约 5 分钟）

  • 列出所有需要跳转的源 URL（secondary pages），以及各自对应的目标 URL（keeper page）。
  • 确认每个目标 URL 是"被保留的页"而不是"新建的页"。已有排名的 URL 作为目标是首选。
  • 确认没有 circular redirect（目标页又 redirect 回源页）。
  • 如果有多个源 URL 跳向同一个目标 URL，确认这种多对一跳转是预期行为。

第二步｜Redirect 响应码验证（约 5 分钟）

使用浏览器开发者工具或 curl -I [源URL] 验证响应头：

  • 每个源 URL 必须返回 301 Moved Permanently 或 308 Permanent Redirect。
  • 禁止使用 302 Found 或 307 Temporary Redirect——这两种状态码告诉 Google"这是临时跳转"，不会转移权重，等于做了 redirect 但没拿到 SEO 价值。
  • 确认响应头里有 Location: [目标URL]，且目标 URL 是完整的绝对地址。
  • 如果服务器配置用的是 302 但业务逻辑是永久跳转，优先改为 301。临时 vs 永久的选择在 redirect QA 里是最容易出错的第一步。

第三步｜Redirect Chain 检查（约 5 分钟）

Redirect chain（重定向链）指源 URL 跳到中间页，再跳到目标页。每多一跳，Google 传递的权重都会打折扣。

  • 对每个源 URL，验证"源 URL → 最终目标 URL"之间是否有中间跳转。
  • 如果 chain 存在（1 hop 以上），优先把源 URL 直接跳到最终目标，减少中间跳转。
  • 常见的 chain 原因：HTTPS → HTTP → HTTPS，或 www → non-www → 另一个域名。先识别是否有服务器层 redirect 再到自己写的 redirect。
  • 如果一个 URL 已经有跳转，但你需要修改目标，不要在旧 redirect 基础上叠加新 redirect，而是直接替换目标。

第四步｜Canonical 验证（约 5 分钟）

Canonical 标签告诉 Google"哪个 URL 是正版"。如果 keeper page 的 canonical 指向了被合并的旧页，Google 会认为旧页才是正版，导致跳转和 canonical 自相矛盾。

  • 打开 keeper page（目标 URL），检查 <head> 里是否有 <link rel="canonical" href="[keeper-page URL]" />。
  • Canonical 必须指向 keeper page 自身。如果指向了旧 URL 或指向了其他版本（HTTP vs HTTPS），立即修正。
  • 如果 keeper page 有多个版本（HTTP / HTTPS / www / non-www），确认 canonical 指向你最希望 Google 收录的那个版本。
  • 如果 keeper page 有 UTM 参数页面，canonical 应指向无参数的版本，避免参数页面和正文页面竞争排名。

第五步｜提交后观察（约 10 分钟 + 等待）

  • 在 Google Search Console 的"网址检查"工具里，逐个验证源 URL 是否已被 Google 识别为 301 跳转。
  • 如果 Google 显示"未收录"或"已识别但未索引"，在 重新索引请求清单 里走一遍是否需要主动提交。
  • 提交后预期：Semrush 指出 Google 通常需要 weeks 到 months 才能完全识别 redirect 并更新排名。不要在提交后 48 小时内预期排名变化。
  • 在 GSC 里设置排名监控：4 周内持续观察 keeper page 的排名是否稳定、是否有下滑趋势。
  • 如果出现排名下滑，检查是否 keeper page 的内容深度不如旧页（301 不会传递内容质量，只会传递权重数字）。

第六步｜判断：哪些情况不该 redirect，应该直接 404

不是所有被合并的页都适合 redirect。直接 404 在某些情况下比 redirect 到一个不太相关的 keeper page 更有利于 SEO。

  • 源 URL 和 keeper page 主题完全不相关——跳转到一个用户不期待到达的页面，体验和 SEO 都会受损。
  • 源 URL 本身已经被惩罚——301 会把这个惩罚一起带过去，先查源 URL 有没有 penalty 再决定是否跳转。
  • 源 URL 有大量自然外链，但 keeper page 内容单薄——权重过去了但内容不够，Google 会发现页面质量下滑。
  • 源 URL 已经 404 多时、外链也已失效——redirect 已经无法借力，直接 404 反而更干净。
  • 源 URL 的关键词和 keeper page 的核心词不在同一语义簇——强行 redirect 会稀释 keeper page 的主题集中度。

5. Redirect 合并后最容易做错的 4 件事

5.1 用 302 代替 301，以为"反正用户感觉一样"

302 和 301 对用户确实感觉一样，但 Google 对它们的处理完全不同。302 是"临时移动"，Google 会保留旧页的索引，不把权重转移到新页。如果你的合并是永久性的，必须用 301 或 308。这个错误在"先临时跳一下，以后再改"的心态里特别常见。

5.2 以为 redirect 上线了，权重就自动传过去了

Semrush 特别指出：Google 需要 weeks 到 months 才能完全处理 redirect 并更新排名信号。如果你在 redirect 上线 3 天后看排名，发现 keeper page 排名掉了，这不代表 redirect 失败了——可能只是 Google 还在处理。多跳 chain 会显著延长这个过程。

5.3 跳转了但 canonical 没有同步更新

这是最容易在 keeper page 上发生的隐性错误。跳转本身是对的，但 keeper page 的 canonical 如果还指向旧 URL，Google 会认为旧 URL 才是正版，权重还是会留在旧 URL。跳转和 canonical 必须配合使用。

5.4 把不相关的旧页 redirect 到 keeper page，以为"权重不浪费"

如果源 URL 和 keeper page 主题不相关，跳转会稀释 keeper page 的主题集中度。Google 的天生的"主题权威性"会被分散到不相关的词上，反而拉低 keeper page 在核心词上的排名。

6. KPI、风险与最小执行建议

Merge Redirect QA Checklist 的价值不在于"做完了检查"，而在于"确认跳转真的有效"。建议追的指标分为三层：

建议 KPI（按 4~8 周追）

  • 曝光量级：合并完成后，keeper page 的 impressions 争取在 4 周内恢复到合并前水平；如果 6 周仍未恢复，优先检查是否有 redirect chain 或 canonical 错误。
  • 预期点击率：keeper page 的 CTR 争取维持合并前水平；若下滑超过 10%，优先检查 meta title 和 description 是否仍然匹配目标关键词。
  • 转化率：从 keeper page 到 联系页 的点击，争取维持合并前 secondary pages 的综合转化水平；若下降明显，优先检查 keeper page 的内容深度和 CTA 是否足够。

本方案风险

  • 本方案风险：用 302 代替 301 导致权重永久无法转移。若发现使用了 302，改为 301 后 Google 需要重新处理，这个过程本身会有 2~4 周的权重损失窗口期。
  • 本方案风险：跳转完成但没有走 canonical 验证，导致 Google 仍然把旧 URL 当正版。这种情况最容易被忽略但代价最大——每次 Google 抓取都是在强化旧 URL 的索引。
  • 本方案风险：合并后没有持续观察，等到月度复盘才发现 keeper page 排名跌了，此时损失已经积累。Redirect QA 不是一次性动作，提交后至少 4 周每周看一次 GSC。

如果你的时间有限，这个 checklist 可以压缩成最核心的 5 步：

最小 Merge Redirect QA 执行表

  • ① 确认用的是 301/308（不是 302/307）
  • ② 确认没有 redirect chain（源 → 目标，最多 1 跳）
  • ③ 确认 keeper page canonical 指向自己
  • ④ Google Search Console 逐个验证（URL 检查工具）
  • ⑤ 4 周内每周看一次 GSC 排名（观察 keeper page 是否稳定）

这 5 步不需要每次全部走完，但每一步都不应该被跳过。当你形成习惯之后，每次合并 redirect 都能在 20 分钟内完成 QA，而不是上线了就不管。
