01

OpenClaw 是自托管 Gateway

官网公开说明把 OpenClaw定义为 self-hosted gateway，可在自己的机器或服务器上运行，用来连接消息渠道、管理会话、路由 Agent，并通过控制界面做配置与查看状态。

02

基础起步需要 Node 22+

Getting Started 页面明确写到 Node 22 或更新版本，并给出 openclaw onboard --install-daemon、openclaw gateway status、openclaw dashboard 这条基础起步路径。

03

Feishu 接入走企业应用与 WebSocket 订阅

Feishu 渠道页公开说明可通过企业应用、权限配置、Bot 能力和 WebSocket 事件订阅接消息，不必暴露公网 webhook，这对内部协同类场景更友好。

哪些潍坊场景更适合先接飞书工作流

本地服务团队的咨询分发

当咨询来自官网、社群或多个人员对接时，最先要做的是把线索统一进一个入口，再做提醒与跟进。

一人公司的内容协同

如果你既要写内容、又要回客户、还要管资料，飞书接入更适合先承接通知、草稿流转与素材整理。

轻团队的任务提醒与资料同步

Agent 不一定先替你做复杂决策，但很适合先做消息转发、状态提醒、资料归档和 FAQ 初步回复。

Workflow

潍坊企业接 OpenClaw 飞书，建议按这 4 步推进

顺序比功能清单更重要。先确认基础能跑，再接最值钱的一条业务流程，才更符合 BUMA 的落地思路。

第一步：先把 Gateway 跑通

先按公开文档完成安装、onboard 和 gateway 状态检查，确认控制界面可用。底座没跑稳，后面所有飞书接入都会变成反复返工。

第二步：只接一个飞书场景

不要一开始把私聊、群聊、审批、文档、全员提醒都上齐。先选一个最频繁、最痛的场景，比如咨询分发或内部提醒。

第三步：给 Agent 明确岗位边界

内容 Agent 负责草稿和整理，协同 Agent 负责提醒和转发，人工负责确认与关键回复。边界清楚，口径才稳，风险也更低。

第四步：围绕真实复盘迭代

当第一条飞书流程稳定后，再增加 FAQ、知识库、更多渠道或更多岗位 Agent，而不是一开始追求“大而全”。

What To Avoid

潍坊团队做飞书接入时，最容易踩的 3 个坑

这些不是耸人听闻，而是从公开文档能反推出来的常见误区。

一上来就接太多入口

入口一多，权限、口径和排查难度都会迅速上升，最容易把试点做成负担。

没定义人工确认边界

涉及报价、承诺、客户关系的内容仍要留人工把关，Agent 更适合先做补位，而不是完全替代。

只看功能，不看维护节奏

真正长期有效的，不是功能越多越好，而是流程有人维护、规则能沉淀、团队愿意持续用。
