Public Facts

先确认公开资料里已经明确的事实

01

OpenClaw 是自托管 Gateway

官方文档把它定义为 self-hosted gateway，可连接 WhatsApp、Telegram、Discord、Feishu 等聊天渠道，并作为会话和路由的控制层。

02

支持 onboarding 与控制界面

公开入门资料提到可通过 `openclaw onboard` 完成向导式配置，也可通过 Control UI 或 dashboard 查看状态、聊天与会话。

03

Feishu 属于可接入通道之一

GitHub 与文档公开信息都把 Feishu 列入支持范围，但具体权限、消息策略与协同方式，仍需要按你自己的环境去配置。

哪些场景更适合先做

咨询承接与FAQ回复

把常见问题、报价说明、资料索取这类高频重复动作先做成标准回复链路，最容易看到效果。

内容生产与改写

如果你长期做公众号、官网更新、社群内容或销售素材，先接内容流程比一开始追求复杂多 Agent 更实用。

飞书协同与任务分发

当团队已经把飞书作为协作中枢时，把 OpenClaw 接进去，更容易把提醒、整理、转发和岗位分工跑顺。

Deploy Order

青岛团队做 OpenClaw，本地部署建议按这 4 步走

先跑通基础，再接流程；先做高频动作，再扩岗位，这样更符合一人公司和小团队的节奏。

第一步：确认运行环境与 Gateway 状态

公开资料显示 OpenClaw 运行依赖 Node 22+，建议先完成 onboard，确认 gateway status、基础配置和本地控制界面可用，再考虑后续接入。

第二步：只接一条最有价值的消息链路

不要一上来把所有通道都接满。先选最常用的一条，例如飞书、Telegram 或网页咨询承接，把入口做稳比铺开更重要。

第三步：给 Agent 定职责，不给它们乱抢活

内容负责什么、客服负责什么、销售负责什么，哪些要人工确认，哪些可自动流转，最好在部署前就写清楚。

第四步：围绕真实业务持续迭代

部署完成只是起点，后面还要不断优化提示词、消息规则、FAQ、知识库和页面承接，才能把工具真正变成生产力。
